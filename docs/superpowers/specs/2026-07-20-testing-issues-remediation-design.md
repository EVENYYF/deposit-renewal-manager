# 测试问题修复设计

日期：2026-07-20

## 1. 背景与目标

本设计处理测试交接文档记录的 8 个问题。先完成问题 1-7 的代码修复、自动化测试和 APK 构建，再对问题 8 进行荣耀手机真机 Debug 验证。

本轮沿用现有 Flutter、Riverpod、UseCase、Repository 和 Drift 分层，不引入通用页面状态框架，不做无关重构。

## 2. 范围

### 包含

1. Android 通知图标、首次授权和系统通知设置跳转。
2. 续期表单的原存款与本次续期信息分组。
3. 停止续期后的状态时序、重复操作保护和可读错误。
4. 首页与客户管理页下拉刷新。
5. 银行、产品统计分类的独立详情页和查询。
6. 自动化验证、APK 构建和最多三次真机调试尝试。

### 不包含

1. 全局状态管理重构。
2. 新的统计筛选、导出或分页功能。
3. 与上述问题无关的视觉改版。

## 3. 总体方案

按现有分层逐项增强，实施顺序为：

```text
Android 通知
    -> 续期与停止续期
    -> 首页和客户页刷新
    -> 统计分类详情
    -> 全量验证和 APK
    -> 真机 Debug
```

每个单元先补充能够暴露问题的测试，再实现最小修复并运行聚焦测试。四个单元完成后执行全量静态检查、测试和构建。

## 4. Android 通知

### 4.1 图标与初始化

- `AndroidInitializationSettings`、定时通知和每日汇总统一使用资源名 `ic_notification`。
- Android 资源继续使用 `android/app/src/main/res/drawable/ic_notification.xml`。
- 通知插件初始化成功后再读取通知能力；Android 13 及以上未授权时，由现有通知能力控制器触发系统权限请求。
- 初始化失败保留明确阶段信息，使重试能重新执行完整初始化流程。

### 4.2 系统通知设置

- Dart MethodChannel 请求打开通知专页，并接收布尔结果。
- Android 优先使用 `Settings.ACTION_APP_NOTIFICATION_SETTINGS`，携带当前包名。
- 系统不存在可处理 Activity 或启动发生异常时，回退到 `Settings.ACTION_APPLICATION_DETAILS_SETTINGS`。
- UI 根据调用结果显示成功或失败反馈，避免无响应。

### 4.3 测试

- 验证初始化、定时通知和汇总通知使用正确资源名。
- 验证未授权时进入通知权限请求路径。
- Android 平台测试验证通知设置 Intent、包名参数、回退行为和 MethodChannel 返回值。

## 5. 续期与停止续期

### 5.1 续期表单

- 续期模式顶部增加“原存款信息”只读区域，展示银行、产品、金额、利率和到期日。
- 下方为“本次续期信息”编辑区域，新产品、新利率及现有允许字段可编辑。
- 客户身份和源存款身份不可修改。
- 提交继续使用现有 Workflow 和 Repository 事务：校验源记录为 `active`，创建新记录，将源记录标记为 `renewed`，写入续期关联，成功后重排通知。

### 5.2 停止续期

- 先提交数据库 `stopped` 状态，使业务状态不受通知取消耗时或失败影响。
- 用户确认后立即禁用相关操作，并在状态提交成功后从当前可操作列表移除卡片。
- 通知取消异步执行；失败时提示通知同步问题，但不回滚已停止状态。
- 刷新请求不等待通知取消完成。
- `DepositNotActiveException` 转换为“该存款已被处理，请刷新后重试”，不向用户展示内部异常名。

### 5.3 测试

- 验证原存款信息只读、本次续期信息可编辑。
- 验证续期事务的源记录和目标记录状态。
- 验证非 `active` 源记录返回可读错误。
- 验证停止后不能再次续期。
- 验证通知取消失败不阻断 `stopped` 状态提交和页面刷新。

## 6. 首页与客户管理刷新

- 首页使用 `RefreshIndicator` 调用 `DashboardController.retry()`。
- 客户管理页使用 `RefreshIndicator` 调用 `CustomerController.retry()`。
- 客户页刷新保留搜索词、银行筛选和产品筛选。
- 有数据、空状态和错误状态均放入始终可滚动的容器，保证任何状态都能触发下拉刷新。
- 首次加载继续显示现有加载态；手动刷新保留旧数据，不闪成整屏 loading。
- 手动刷新失败时保留旧数据，并显示可见反馈。

测试覆盖有数据和空状态刷新、单次请求、筛选保留、刷新期间旧内容稳定以及失败反馈。

## 7. 统计分类详情

### 7.1 数据边界

- `DepositStatisticsUseCases` 增加按分类加载详情的方法。
- 分类维度使用固定枚举 `bank` 和 `product`；数据层通过显式映射选择 SQL 列，不接受任意列名。
- 查询仅返回有效客户的 `active` 存款。
- 结果按到期日、客户姓名和存款 ID 稳定排序。
- “未填写”分类使用明确的空值或空白条件，不把 UI 文案拼接进 SQL。

详情模型包含客户姓名、手机号、银行、产品、金额、利率数值与精度、到期日和存款 ID。

### 7.2 页面与导航

- 银行和产品统计行增加点击入口。
- 点击后进入独立详情页，页面标题包含分类维度与分类值。
- 页面支持加载、空、错误和手动刷新状态。
- 长列表使用惰性构建，按既定排序展示全部匹配记录。

### 7.3 测试

- 验证维度到 SQL 列的固定映射。
- 验证空值分类、仅 `active` 数据和稳定排序。
- 验证统计行点击导航、标题、字段展示、空态和错误重试。

## 8. 错误处理原则

- 业务状态提交与通知同步分离，通知失败不回滚已经成功的业务变更。
- 并发或过期页面导致的非活动存款操作统一转换为可读提示。
- 刷新失败保留旧数据，首次加载失败才使用整页错误态。
- 平台跳转必须返回明确结果，Android 不支持通知专页时自动回退应用详情页。

## 9. 验证与交付

### 9.1 自动化验证

按以下顺序执行：

1. 聚焦 Dart 单元和组件测试。
2. Android 平台测试。
3. `dart format`。
4. `flutter analyze`。
5. 全量 `flutter test`。
6. Android APK 构建。

只有实际执行成功的项目标记为“已验证”。APK 构建后记录产物路径。

### 9.2 真机验证

1. 最小前置检查：确认 `adb` 可执行、设备已识别并授权。
2. 尝试 Debug 安装、启动和连接。
3. 失败时只针对明确错误调整，最多重试三次，不做无差别重复。
4. 第三次仍失败后立即停止，提供 APK 路径、失败摘要、已尝试动作和手动验收清单，由用户继续验证。
5. 真机检查通知授权、通知设置页、通知触达、下拉刷新、停止续期、统计详情和重启后的数据与通知状态。
6. 未实际完成的真机场景标记为“待用户验证”。

## 10. 完成标准

- 问题 1-7 均有对应实现和回归测试。
- 格式化、静态分析和全量测试通过。
- Android APK 构建成功并报告路径。
- 真机验证遵守最多三次重试限制；成功则记录结果，失败则交付完整的手动验证信息。
